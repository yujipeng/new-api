## 1. 背景与目标

### 1.1 业务背景

tcreditaim（基于 new-api 内核的 LLM 中转平台）已运行用户 → 模型 → 渠道（上游供应商）调用链路并按用户侧 `quota` 计费扣款。当前缺口：

- **没有"成本"视图**：`Log` 表只记录用户侧 `Quota`（收入侧），未记录调用对应的供应商成本；无法核算单次/单渠道/单用户的**毛利**。
- **没有结构化账单**：管理员只能逐条翻 `logs`，无法在用户、渠道、全流程三个粒度上做日/月对账。
- **财务对账 & 业务复盘困难**：当前依赖手工 SQL 汇总，月底对账周期长、口径不一致风险高。

### 1.2 目标

为 tcreditaim 平台**增加账单模块**，提供：

1. **用户账单**（用户和管理员可见）：用户视角的消费明细与汇总。
2. **渠道账单**（管理员可见）：渠道（供应商）视角的成本与利润。
3. **全流程账单**（管理员可见）：单笔调用打穿"用户 → 渠道"的明细行表。

所有账单支持**日维度**与**月维度**两种粒度查询。

### 1.3 北极星指标

| 指标 | 定义 | 数据源 | 目标值 | 时间窗 |
| --- | --- | --- | --- | --- |
| 管理员对账时长 | 月度对账耗时（从打开账单到完成一次完整毛利核对） | 管理员行为埋点（v1 上线后接入） | 上线后从基线 ≥ 60 分钟下降到 ≤ 10 分钟 | 上线后第 1 个完整月 |

### 1.4 护栏指标

| 护栏 | 定义 | 目标 |
| --- | --- | --- |
| 账单数据准确性 | 全流程账单"收入合计 / 成本合计 / 利润"与 `logs` 表逐条 SUM 的差异率 | < 0.1%（容忍浮点取整） |
| 账单查询 P95 延迟 | 一个用户 30 天日账单列表（≤ 30 行）的 API 响应 | < 800ms（数据库无新增索引情况下 ≤ 2s 视为可接受，须在 v1.1 优化） |

## 2. 用户与场景

### 2.1 目标用户

| 角色 | 描述 | 在账单模块中的诉求 |
| --- | --- | --- |
| 普通用户（API 调用方） | 通过 tcreditaim 调用模型的开发者 / 业务方 | 自查日 / 月消费、明细可追溯、可导出做内部报销 |
| 管理员（运营 / 财务） | 平台运营、财务对账人员 | 查全平台用户消费、查每个渠道成本与利润、定位异常调用 |

### 2.2 核心场景

#### 场景 A：用户自助查询消费（普通用户）

- **现状痛点**：当前只能进 `Log` 页面看流水，没有"今天我花了多少"、"上个月总账"的聚合视图。
- **期望改善**：进入"我的账单"即可看到日 / 月汇总卡片 + 按分组（`User.Group`）拆分的子表 + 可导出 CSV。

#### 场景 B：财务月度对账（管理员）

- **现状痛点**：现在月底需写 SQL 拼接 `logs` + `users` + `channels`，再下载 Excel 手工分组，单次耗时 ≥ 60 分钟，跨人工易出错。
- **期望改善**：进入"渠道账单 / 全流程账单"，选月份，得到结构化收入 / 成本 / 利润表，一键导出 CSV，目标 ≤ 10 分钟。

#### 场景 C：渠道毛利分析（管理员）

- **现状痛点**：当前无法回答"渠道 X 这个月赚没赚钱"。
- **期望改善**：渠道账单页按日 / 月给出每个渠道的"调用量 / 收入 / 成本 / 利润 / 利润率"。

### 2.3 数据 / 证据来源

- **现状证据**：上游 PJM（[@项目管理专家](mention://agent/4c397204-2e4f-4a1e-a878-d2df27e7a9d3)）在 issue [TES-28](mention://issue/26ce481d-8463-42de-ad03-39f6ad59686a) 描述中明确给出需求字段与三类账单结构，并于 2026-05-20 12:04 确认四项关键决策（见第 4 节）。
- **代码现状**（基于本仓库 `model/` 目录 2026-05-20 巡查）：
  - `model/log.go` 已含 `UserId / ChannelId / ModelName / PromptTokens / CompletionTokens / Quota / CreatedAt / Type / Group`。
  - `model/pricing.go` 含 `ModelRatio / ModelPrice / CompletionRatio / CacheRatio / QuotaType / BillingMode`（**售价侧**）。
  - `model/channel.go` 含 `Group / Type / Balance / UsedQuota / Priority` 等，**无成本字段**。
  - `model/user.go` 含 `Group / Quota / UsedQuota`。
  - 全仓库未搜索到任何 `cost / 成本 / supplier_price` 字段 → 成本数据**必须新增**（见第 4 节决策 1）。

> 备注：v1 上线前**没有**对账时长基线测量数据；上线时埋点必须同步落地，以便上线后第 1 个完整月可计算指标。基线获取见第 11 节待澄清项。

## 3. 候选方案

### 方案 A：物化 + 增量（推荐 MVP 默认）

> **决策已锁定（PJM 2026-05-20 12:04 确认）**：成本口径 A、月聚合 A、分组按 User.Group / Channel.Group、MVP = A+B。本方案与确认完全一致，列出仅作记录与权衡说明。

- **数据模型**：
  - 新增 `channel_pricing` 表（成本定价配置，与 `pricing` 售价表对称）。
  - 新增 `bill_daily_full` 表（全流程日账单，每行 = 日期 × 用户 × 用户分组 × 渠道 × 渠道分组 × 模型 的汇总）。
  - **用户日账单 / 渠道日账单 / 月账单**：**不物化**，按需在 `bill_daily_full` 上 SUM。
- **写入路径**：日终离线 job（凌晨 02:00）扫描前一天 `logs` + `channel_pricing` 计算成本，回填 `bill_daily_full`。
- **用户价值**：满足所有 MVP 场景；查询稳定；后续可在此基础上加调账（v2）/告警（v2）。
- **商业价值**：高 — 直接缩短月度对账时间。
- **复杂度**：中（一张物化表 + 一个 cron job + 三组查询接口）。
- **放弃它的代价**：无替代方案能在不破坏现有调用链路 / 不引入实时成本计算开销的前提下满足 MVP。

### 方案 B：实时计算（不推荐）

- 不物化，所有账单查询都 JOIN `logs` × `channels` × `channel_pricing` 在线聚合。
- **用户价值**：等价。
- **商业价值**：等价但风险高。
- **复杂度**：低（无新表），但查询性能差，30 天范围聚合在 logs 千万级量下 P95 ≥ 数秒。
- **放弃它的代价**：低；可作 v0.5 灰度试探，但 GA 必须切到方案 A。
- **不选原因**：违反护栏指标 2（P95 延迟）。

### 方案 C：写入路径同步落库（不推荐）

- 在 `RecordConsumeLog` 写 `logs` 的同时按当时 `channel_pricing` 计算成本并写到 `bill_daily_full`。
- **优势**：实时，无 cron 滞后。
- **劣势**：① 与计费扣款共事务，失败回滚成本高；② 成本定价变更（追溯调价）无法回填；③ 增加热路径 RT。
- **放弃它的代价**：低，方案 A 的 T+1 时延对账单业务足够。
- **不选原因**：风险 > 收益；调价追溯能力是必须的（财务常见诉求）。

## 4. 推荐方案与理由

**推荐：方案 A（物化 + 增量）**——与 PJM 2026-05-20 12:04 锁定的四项决策完全一致。

| 维度 | 方案 A | 方案 B | 方案 C |
| --- | --- | --- | --- |
| 业务价值 | 9/10（高频对账强诉求） | 9/10 | 9/10 |
| 技术成本 | 6/10（中等：1 张表 + 1 cron + 3 套查询） | 4/10（看似低，但有查询性能优化债） | 7/10（侵入热路径） |
| 用户影响（管理员 + 普通用户） | 9/10 | 6/10（慢） | 9/10 |
| **综合** | **8** | **6.3** | **8.3 → 折算后 6**（热路径风险扣分） |

四项锁定决策映射：

1. **成本数据口径 = A**：新增 `channel_pricing` 表（与 `pricing` 售价表对称结构），按"成本定价 × tokens"计算单次成本写入 `bill_daily_full.cost`。
2. **月账单聚合 = A**：月账单 = 该月 `bill_daily_full` 简单 SUM，不独立写入，不支持月度调账（v2 再考虑）。
3. **分组字段 = 确认**：
   - 用户分组 = `User.Group`（注：`Log.Group` 已经在写入时落了 `User.Group` 快照，沿用即可，避免历史用户改组导致回看变化）。
   - 供应商分组 = `Channel.Group`（CSV 字段；首版**仅取第一个分组**作为账单口径，后续若有多分组诉求再迭代，见第 11 节）。
4. **MVP 范围 = A + B**：查询 / 列表 / 导出 + 日/月切换 + 时间范围筛选；C 调账、D 告警 → 留 v2。

## 5. 功能详述

### 5.1 用户故事

#### US-1：普通用户查看自己的日 / 月账单

> 作为**普通用户**，我希望**在「我的账单」页面看到我每日 / 每月的总消费 + 按用户分组与模型拆分的明细 + CSV 导出**，以便我**自查消费 & 做团队内部报销**。

#### US-2：管理员查看全用户的账单

> 作为**管理员**，我希望**按用户、按时间筛选所有用户账单（日 / 月）**，以便我**响应客户对账咨询 & 月度收入复核**。

#### US-3：管理员查看渠道账单

> 作为**管理员**，我希望**按渠道、按时间看到调用量 / 成本 / 收入 / 利润**，以便我**评估渠道毛利、决定渠道权重调整**。

#### US-4：管理员查看全流程账单

> 作为**管理员**，我希望**看到一张 `日期 - 用户 - 用户分组 - 调用量 - 收入 - 渠道分组 - 渠道 - 调用量 - 成本 - 利润` 的明细表（每行一种维度组合）**，以便我**做异常归因（哪个渠道在哪天给哪个用户带来了负利润）**。

#### US-5：管理员配置渠道成本定价

> 作为**管理员**，我希望**在渠道设置页可以为该渠道下每个模型配置成本定价（与售价同结构）**，以便**账单系统能正确计算成本**。

### 5.2 验收标准（Given / When / Then）

#### AC-1：用户日账单

- **Given** 用户 `U1` 在 2026-05-19 调用了模型 `M1` 共 100 次，对应 `Quota=1000`
- **When** 用户访问"我的账单 → 日"并筛选日期 = 2026-05-19
- **Then** 页面显示当日"调用量 100 / 消费（Quota）1000"汇总卡，明细行 ≥ 1 行（按分组 × 模型），允许导出 CSV

#### AC-2：用户月账单（SUM）

- **Given** 用户 `U1` 在 2026-05 月内每天有日账单
- **When** 用户切换粒度到"月"并选择 2026-05
- **Then** 页面显示 2026-05 全月汇总，数值 = 该月所有日账单的简单 SUM；不存在独立的"月账单流水"

#### AC-3：渠道账单成本计算

- **Given** 渠道 `C1` 的成本定价为 `M1=0.001 USD/1K tokens`，用户 `U1` 当日通过 `C1` 调用 `M1` 共 5000 tokens
- **When** 日终 job 在凌晨 02:00 执行
- **Then** `bill_daily_full` 出现一行：`channel_id=C1, cost=0.005 USD（按 Quota 单位换算后存储为整数 quota，规则同现有体系）, profit = revenue - cost`

#### AC-4：成本定价缺失的兜底

- **Given** 渠道 `C2` 未配置任何成本定价
- **When** 日终 job 处理含 `C2` 的调用记录
- **Then** 该行 `cost = 0`、`profit = revenue`，并打一条 WARN 日志 + 在管理员视图标记"⚠️ 未配置成本"（不阻断写入）

#### AC-5：管理员筛选

- **Given** 平台存在 ≥ 100 个用户、≥ 20 个渠道
- **When** 管理员进入"渠道账单"，选时间范围 2026-05-01 ~ 2026-05-31，按"渠道分组 = group-A"筛选
- **Then** 页面在 P95 ≤ 2 秒内返回（30 天 × 20 渠道 × 多模型 ≤ 数千行），支持 CSV 导出

#### AC-6：CSV 导出格式

- **Given** 任意账单视图当前已显示数据
- **When** 用户点击"导出 CSV"
- **Then** 下载文件，列头与表头完全一致，编码 UTF-8 with BOM（Excel 直接打开中文不乱码），数值不带千分符

#### AC-7：权限隔离

- **Given** 普通用户 `U1` 登录
- **When** `U1` 尝试访问"渠道账单"或"全流程账单"接口
- **Then** 接口返回 403；前端不展示菜单项

#### AC-8：数据准确性

- **Given** 任意一天的 `bill_daily_full`
- **When** 与同日 `logs` 表直接 SUM 对比
- **Then** "收入合计"差异率 < 0.1%；不一致由对账脚本（v1.1 提供）报警

### 5.3 范围内 / 范围外

| 范围内（MVP） | 范围外（v2 及以后） |
| --- | --- |
| 日 / 月账单查询 + 列表 + 导出 | 管理员手动调账 |
| 三种账单视图（用户 / 渠道 / 全流程） | 异常告警（成本异常波动、利润为负自动报警） |
| 时间范围 + 分组 + 渠道筛选 | 月账单作为独立流水（支持调账、退款、补录） |
| 渠道成本定价配置（管理员） | 多币种 / 汇率管理 |
| 日终离线 job（T+1） | 准实时账单（分钟级） |
| 准确性自检脚本（v1.1） | 渠道多分组（CSV 中每个分组独立成行） |

## 6. 指标定义

### 6.1 北极星指标

- **管理员对账时长**
  - 定义：管理员从打开账单页面到完成一次完整对账操作（输出 CSV / 关闭页面）的时长 P50。
  - 数据源：前端埋点 `billing.session.start / end`（v1 上线必交付）。
  - 目标：上线后第 1 个完整月 ≤ 10 分钟（基线 ≥ 60 分钟，需在上线前一周补一次基线测量，方法见第 11 节待澄清项）。
  - 时间窗：上线 +30 天。

### 6.2 护栏指标

- **账单数据准确性**
  - 定义：`SUM(bill_daily_full.revenue) / SUM(logs.quota) - 1` 的绝对值（限同日范围）。
  - 数据源：对账脚本（v1.1 提供）每日 03:00 跑。
  - 目标：< 0.1%。
  - 时间窗：日级别。

- **账单查询 P95 延迟**
  - 定义：账单列表 API 在 30 天范围请求下的 P95 响应时间。
  - 数据源：APM 监控（沿用现有 Prometheus / NewRelic，具体待运维同步，见第 11 节）。
  - 目标：< 800ms（短期 ≤ 2s 可接受，v1.1 必须达标）。
  - 时间窗：上线后逐周观测。

## 7. 合规与监管

- **数据范围**：账单仅存储已在 `logs` / `users` / `channels` 中的字段聚合，**不引入新的个人敏感信息**。
- **GDPR / 数据保留**：账单数据保留策略与 `logs` 一致（沿用现有清理策略；具体年限由平台运营定，本 PRD 不重定义）。
- **多租户数据隔离**：用户账单严格按 `user_id` 过滤；普通用户接口必须服务端鉴权（参考现有 `middleware/auth`）。
- **导出审计**：管理员 CSV 导出操作需记入操作日志（沿用现有 `operation_log`，无需新增）。
- **法务确认**：本 MVP 不涉及新增数据采集 / 跨境传输 / 第三方共享，**判定无需独立法务签字**，由架构师在 design.md 阶段最终复核；如架构师有异议请在第 8 节"跨团队对齐"中反馈，本节据此 v2 版本化。

> 注：tcreditaim 目前的部署 / 客户地域分布未在本 PRD 范围内重新调研。若客户群体覆盖海外（含 EU），架构师需在 design.md 中标注 GDPR 适用性结论。

## 9. 风险与回退

### 9.1 用户假设风险

| 风险 | 概率 | 影响 | 缓解 |
| --- | --- | --- | --- |
| "管理员对账耗时基线 ≥ 60 分钟"未量化即作为目标 | 中 | 高（指标不可证伪） | 上线前一周补测一次基线（第 11 节列为待澄清项 #1） |
| 渠道多分组拆账诉求被忽略 | 中 | 中（财务可能要求按"渠道 × 分组"独立行） | MVP 仅取 `Channel.Group` 第一个分组；与运营在评审会确认是否够用 |
| 用户对"日账单 = 自然日（UTC / 本地时区）"理解不一致 | 中 | 中 | 在 design.md 中**明确日切时区 = 系统统一时区**（建议 Asia/Shanghai，由架构师评估服务器时区） |

### 9.2 商业风险

- **成本定价误配 → 利润误算**：兜底 = AC-4（缺失成本时 cost=0 并打 WARN）；上线后需运营在 1 周内逐渠道补齐成本定价。
- **导出 CSV 被滥用泄露财务数据**：依托现有权限体系 + 操作日志即可；不在 PRD 引入额外水印。

### 9.3 回退方案

- 离线 job 失败 → 不影响在线业务（账单视图显示"昨日数据生成中"占位）；运营手动重跑 job 即可。
- `bill_daily_full` 数据错误（如成本定价错配）→ truncate 当日分区数据并重跑 job（无需回滚业务表）。
- 整个账单模块下线 → 前端隐藏菜单 + 接口返回 503；不影响计费扣款（与扣款路径完全解耦）。

## 附录 A：数据模型示意（仅为 PRD 沟通用，最终以 design.md 为准）

```
channel_pricing
├── id
├── channel_id  (FK channels.id)
├── model_name
├── model_ratio / model_price / completion_ratio / ...  // 结构与 pricing 表对称
├── currency  (default USD)
└── updated_at

bill_daily_full   // 一行 = 一天 × 用户 × 用户分组 × 渠道 × 渠道分组 × 模型
├── stat_date         // YYYY-MM-DD
├── user_id
├── user_group        // 取 Log.Group（用户历史分组快照）
├── channel_id
├── channel_group     // 取 Channel.Group 首个分组（MVP 简化）
├── model_name
├── request_count     // 调用量
├── revenue_quota     // 收入（与 Log.Quota 体系一致）
├── cost_quota        // 成本（按 channel_pricing 计算后换算为 quota）
├── profit_quota      // revenue - cost
└── generated_at
```

**用户日账单 / 渠道日账单 / 月账单**：均在 `bill_daily_full` 上按粒度 GROUP BY，**不物化**。
