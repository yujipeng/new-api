# 额度异常排查 SOP

> 适用版本：v1.0.0+　｜　最后核对：2026-05-20　｜　代码基线：`relay/helper/price.go`、`service/text_quota.go`

## 目标

当管理员或用户反馈"扣费不符合预期"时，按本 SOP 定位根因并修复。

## 前置条件

- 拥有管理员权限（可查看日志、用户额度、渠道配置）
- 了解 [计费公式](../01-concepts/billing-formula.md) 中三种"优先级"语义

---

## 常见症状与排查路径

### 症状 1：扣费为 0（应该扣费但没扣）

| 排查步骤 | 操作 | 预期 |
|----------|------|------|
| 1. 检查 GroupRatio | 系统设置 → 计费设置 → 分组定价，查看该用户所在分组的倍率 | 若 `GroupRatio = 0`，则为免费分组，扣费为 0 是正常行为 |
| 2. 检查 EnableFreeModelPreConsume | 系统设置 → 计费设置 → 额度配置 | 若关闭，`GroupRatio=0` 时预扣额度强制为 0 |
| 3. 检查 TotalTokens | 查看该请求的日志记录 | 若上游返回 `TotalTokens=0`，系统强制 `quota=0` |

**修复动作**：
- 若分组倍率误设为 0 → 修改分组倍率为正确值
- 若上游未返回 token 数 → 检查渠道适配器是否正确解析 usage

---

### 症状 2：扣费远高于预期

| 排查步骤 | 操作 | 预期 |
|----------|------|------|
| 1. 确认计费分支 | 查看日志中 `usePrice` 字段 | `true` = 按次计费；`false` = 按 token 计费 |
| 2. 按次计费过高 | 检查 ModelPrice 配置 | 确认单价是否合理（单位：$/次） |
| 3. 按 token 计费过高 | 检查 ModelRatio × GroupRatio 乘积 | 两者相乘，任一过大都会导致高额扣费 |
| 4. 检查 GroupGroupRatio | 系统设置 → 计费设置 → 分组定价 → 特殊倍率 | 特殊倍率会覆盖普通 GroupRatio，可能被设为高值 |
| 5. 检查 OtherRatios | 查看日志中附加倍率列表 | 所有 OtherRatios 逐一相乘，累积效应可能很大 |
| 6. 检查 CompletionRatio | 系统设置 → 计费设置 → 模型定价 | 某些模型补全倍率较高（如 o1 系列思考 token） |

**修复动作**：
- 调整对应倍率至合理值
- 若 GroupGroupRatio 误配 → 删除或修正特殊倍率条目

---

### 症状 3：扣费略低于预期（缓存折扣未生效）

| 排查步骤 | 操作 | 预期 |
|----------|------|------|
| 1. 检查 CacheRatio 配置 | 系统设置 → 计费设置 → 模型定价 → 缓存倍率 | 应 < 1.0（如 0.1 表示缓存命中按 10% 计费） |
| 2. 检查上游是否返回 cache_tokens | 查看请求日志中 `CacheTokens` 字段 | 若为 0，说明上游未返回缓存信息 |
| 3. 检查渠道是否支持缓存报告 | 确认渠道类型（如 Claude 支持，部分 OpenAI 兼容渠道不支持） | — |

**修复动作**：
- 确认上游确实支持并返回了缓存 token 数
- 配置正确的 CacheRatio（通常 0.1 ~ 0.5）

---

### 症状 4：模型价格未配置错误

错误信息：`模型 X 的价格未配置`

| 排查步骤 | 操作 | 预期 |
|----------|------|------|
| 1. 检查自用模式 | 系统设置 → 运营设置 | 自用模式开启时会使用默认倍率 37.5，不会报此错 |
| 2. 检查 ModelRatio 配置 | 系统设置 → 计费设置 → 模型定价 | 确认该模型名称已配置倍率 |
| 3. 检查模型名称匹配 | 对比请求中的 model 字段与配置中的模型名 | 大小写、前缀需完全匹配 |

**修复动作**：
- 在模型定价中添加该模型的 ModelRatio 或 ModelPrice
- 或开启自用模式（仅限内部使用场景）

---

### 症状 5：不同用户相同请求扣费不同

| 排查步骤 | 操作 | 预期 |
|----------|------|------|
| 1. 对比用户分组 | 管理面板 → 用户管理，查看两用户所属分组 | 不同分组有不同 GroupRatio |
| 2. 检查 GroupGroupRatio | 确认是否存在用户组→目标组的特殊倍率 | 特殊倍率覆盖普通倍率 |
| 3. 检查 auto_group | 确认是否启用了自动分组切换 | 请求上下文可能注入不同的计费分组 |

**修复动作**：
- 若预期一致 → 统一分组倍率配置
- 若预期不同 → 确认 GroupGroupRatio 设置正确反映业务意图

---

## 通用排查工具

| 工具 | 用途 |
|------|------|
| 请求日志 | 查看单次请求的 token 数、倍率、最终 quota |
| 用户额度页 | 查看余额变动历史 |
| 系统设置 → 计费设置 | 集中查看所有倍率配置 |
| `pkg/billingexpr/expr.md` | 表达式计费的语法与调试方法 |

## 升级路径

若按上述步骤排查后仍无法定位问题：
1. 导出该请求的完整日志（含 token 明细）
2. 对照 [计费公式](../01-concepts/billing-formula.md) 手动计算预期值
3. 若手动计算结果与系统一致 → 用户对计费规则理解有误，引导阅读概念文档
4. 若手动计算结果与系统不一致 → 可能是代码 bug，提交 issue 并附上计算过程
